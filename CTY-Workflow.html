Product Vision

What CTY is becoming

CTY is the app that makes you fall in love with New York City — neighborhood by neighborhood, block by block. It's your personal map of the city you're building, a living record of everywhere you've been, everything you've tasted, and every corner you've turned. Not a review site. Not a tourist guide. A city exploration game that rewards curiosity.

The core product is an interactive map that feels as premium as Apple Maps, overlaid with a deep, accurate data layer of restaurants, attractions, and experiences across all five boroughs. Everything else — routes, stamps, social, transit — extends from this foundation.

Core Principles

Map-First

The map is the product. Every feature should enhance the map experience, not distract from it. It should feel alive — like the city itself.

Data Accuracy

Bad data kills trust. Every restaurant, every boundary, every transit line needs to be accurate. We'll use multiple API sources and manual curation.

Exploration as a Game

Achievement-based stamps, completion percentages, route challenges. Exploring NYC should feel like a game you want to 100%.

The City is Alive

The app should feel different at 8am vs 11pm, in the West Village vs the Bronx. Dynamic theming that reflects where and when you are.

Interactive Map Phase 1

The soul of the product — Apple Maps quality, NYC personality
Current State

The map currently uses Leaflet with CartoDB Dark tiles and CSS rotation for vertical Manhattan. This approach has fundamental limitations: CSS rotation creates visual artifacts, Leaflet tiles don't support native bearing rotation, and the dark-only tiles feel lifeless.

Target: Mapbox GL JS

The definitive solution is Mapbox GL JS — which already powers the Explorer page. It supports native bearing rotation (no CSS hacks), custom styles via Mapbox Studio, 3D buildings, smooth animations, and high-DPI rendering. This is how you get Apple Maps quality.

Native Bearing Rotation

Set bearing: 29 in the map config. Manhattan goes vertical. Tiles, labels, and polygons all rotate together cleanly. No CSS transform needed.

Custom Map Style

Design a custom Mapbox Studio style: warm dark base, visible streets, elegant labels, 3D buildings at zoom 15+. The map IS the brand.

Neighborhood Polygons

GeoJSON source with fill-extrusion layers. Invisible by default, colored on visited/lived. Smooth opacity animations on hover and state change.

POI Markers

Custom markers for restaurants and attractions that appear at appropriate zoom levels. Clustered at far zoom, individual at close zoom.

Implementation Steps
1

Replace Leaflet with Mapbox GL

Swap out the Leaflet map initialization, tile layer, and polygon rendering with Mapbox GL equivalents. The Explorer page already has a Mapbox setup to reference. Use bearing: 29 and pitch: 0 for the initial view.

2

Design Custom Map Style in Mapbox Studio

Fork the "Dark" base style. Warm up the background colors (deep navy instead of pure black), increase road visibility, add warm label colors. Enable 3D buildings. Export as a style URL.

3

Convert Neighborhood Polygons to GeoJSON Source

Convert LOCALITY_BOUNDARIES into proper GeoJSON FeatureCollection. Add as a Mapbox source with fill and fill-outline layers. Drive visibility from feature-state for visited/lived status.

4

Implement Hover & Click Interactions

Use Mapbox's queryRenderedFeatures for hover detection. Show tooltip with neighborhood name and status. Click opens the detail panel. Smooth transitions between states.

5

Add POI Layer (After Data Pipeline)

Once restaurant/attraction data is enriched, add as clustered GeoJSON source. Custom icons per category. Filter by zoom level and category. Tap to open spot detail.

Tech Stack
Mapbox GL JS v3 Mapbox Studio GeoJSON Feature State API Supercluster
Key Decisions
Free tier vs paid Mapbox?
Free tier: 50,000 map loads/month — more than enough for launch.
Mapbox free tier includes Studio, GL JS, and geocoding. Only pay when you scale past 50K monthly active users.
Vertical Manhattan: CSS rotation vs native bearing?
Native bearing (Mapbox GL). Eliminates all CSS rotation artifacts.
Mapbox GL renders rotated tiles, labels, and polygons natively. No container overflow hacks, no sideways text, no clipped corners.

Data Pipeline Phase 1

Sourcing, enriching, and maintaining accurate NYC data
Strategy: Multi-Source Aggregation

No single API has complete, accurate NYC data. The strategy is to combine multiple sources — Google Places, Yelp, and Foursquare — cross-reference for accuracy, then layer in manual curation for the "best of" selections. Start with top spots per neighborhood, expand to comprehensive coverage over time.

Data Sources
SourceStrengthsCostUse For
Google Places API Most complete POI data, real-time hours, photos, ratings $17/1000 detail requests (free $200/mo credit) Primary data source — names, locations, hours, photos
Yelp Fusion API Strong restaurant ratings, price levels, review counts Free: 5000 calls/day Restaurant ratings, price data, category classification
Foursquare Places API Good nightlife/attraction data, tips, popularity metrics Free: 100K calls/month Attractions, bars, supplementary categories
NYC Open Data Official boundaries, permits, public facilities Free Neighborhood boundaries, parks, public landmarks
Manual Curation Quality control, "best of" picks, local knowledge Time Featured spots, editorial picks, accuracy overrides
Borough-by-Borough Collection Strategy

Process each borough's neighborhoods systematically. For each neighborhood, the pipeline runs: 1) Query all three APIs for the area's bounding box, 2) Deduplicate by name+location proximity, 3) Merge best data from each source, 4) Classify into CTY categories, 5) Manual review of top picks.

1

Build the Collection Script

Node.js script that takes a neighborhood name + bounding box, queries all three APIs, deduplicates, merges, and outputs a JSON file per neighborhood. Rate-limited to stay within free tiers.

2

Start with Manhattan (62 neighborhoods)

Manhattan has the most detailed boundary data and the densest POI coverage. Run collection for all 62 neighborhoods. Manual review for top 10 spots per neighborhood.

3

Expand: Brooklyn → Queens → Bronx → Staten Island

Process each borough in order of data density. Brooklyn will be nearly as rich as Manhattan. Outer boroughs may need more manual curation to hit quality targets.

4

Set Up Refresh Pipeline

Restaurants open and close constantly. Build a weekly cron job that re-queries APIs for changes — new openings, permanent closures, updated hours. Flag changes for review.

Data Schema (Per Spot)

Core Fields

id, name, lat, lng, neighborhood_id, borough, category, subcategory, address, phone, website

Enrichment Fields

google_rating, yelp_rating, foursquare_rating, price_level, hours, photo_urls[], is_curated, feature_tags[]

CTY Fields

cty_score (aggregated), visit_count, avg_user_rating, is_featured, stamp_eligible, route_ids[]

Metadata

created_at, updated_at, source_ids{}, last_verified, status (active/closed/seasonal)

Tech Stack
Node.js Collection Scripts Google Places API Yelp Fusion API Foursquare API Supabase (PostgreSQL + PostGIS) GitHub Actions (cron refresh)

Theme & Design System Phase 1

Dynamic theming that makes the city feel alive

The app shifts its personality throughout the day and based on where you're looking. Morning exploration feels crisp and bright. Evening browsing feels warm and intimate. Implementation Approach
1

CSS Custom Properties Theme Engine

All colors defined as CSS variables on :root. A JavaScript function runs on load and every hour, checking time of day and updating variables. Borough context shifts accent variables when the user is viewing a specific borough.

2

Mapbox Studio: 5 Map Style Variants

Create 5 Mapbox styles (one per time period) or use runtime styling to adjust colors dynamically. The map should feel different at night vs morning — warmer road colors, different building tints, adjusted label brightness.

3

Smooth Transitions

Theme changes should animate — 0.5s CSS transition on all color properties. The map style change can use Mapbox's setStyle with a crossfade. No jarring switches.

Day Plans & Routes Phase 2

Pre-built themes, AI-generated plans, and user-created routes
Three Types of Routes

Curated Routes

Hand-crafted by you. "Best Pizza Walk in Manhattan", "Brooklyn Bridge to DUMBO", "Museum Mile Marathon". High editorial quality. These are the flagship content.

AI-Generated Plans

User says "I have 3 hours in the East Village and love coffee." AI builds an optimized route with timing, transit, and alternatives. Powered by OpenAI + Mapbox Directions.

User-Created Routes

Any user can create and share routes. Pin spots on the map, add notes, set timing. Share via link. Community votes on the best ones. User-generated content engine.

Route Data Model

Each route is a sequence of stops (spots from the database) with timing, notes, and connecting transit segments. Routes belong to a category (food crawl, cultural tour, shopping spree, date night, etc.) and can be favorited, rated, and forked.

Route Schema

Route Object

id, title, description, category, author_id, borough, estimated_duration, distance_miles, stop_count, cover_photo_url, is_curated, is_ai_generated

Stop Object

spot_id, order, estimated_time_minutes, note, transit_to_next (walk/subway/bus), transit_duration_minutes

Multi-Dimension Ratings Phase 2

Rate food, vibe, value, and service — separately
Rating Dimensions

Instead of a single star rating, users rate each dimension separately. This gives much richer data — a place can have incredible food but terrible service, and CTY captures that nuance. Different spot types get different dimensions.

Restaurants

Food (taste & quality), Vibe (ambiance & decor), Value (price vs quality), Service (staff & speed). Each rated 1-5 with half-star increments.

Attractions

Experience (how memorable), Vibe (atmosphere), Value (worth the cost/time), Accessibility (ease of getting there/in). Adapted dimensions for non-food spots.

Rating UX
1

Quick Rate (Low Friction)

After marking a spot as visited, a subtle prompt slides up: "How was it?" User can tap a quick overall score (1-5) or expand for the full multi-dimension form. Never force the full form.

2

Detailed Rating Card

Four sliders (one per dimension) with emoji feedback at each level. Optional text note (up to 280 chars — a "micro review"). Optional photo upload. The whole thing takes 30 seconds max.

3

Aggregate Display

Spot detail shows a radar chart of all four dimensions with the community average. Individual dimension scores visible. "Based on X ratings" for transparency. Your personal rating highlighted if you've rated.

Schema

Rating Object

id, user_id, spot_id, dimension_scores: { food: 4.5, vibe: 3.0, value: 5.0, service: 4.0 }, overall (computed average), note, photo_url, created_at

Stamps & Achievements Phase 2

Gamified exploration — milestones that reward curiosity
Achievement Categories

Explorer Stamps

"First Steps" (visit 1 neighborhood), "Borough Breaker" (visit all 5 boroughs), "Manhattan Master" (visit all Manhattan hoods), "Full Map" (every neighborhood visited)

Foodie Stamps

"Pizza Pilgrim" (5 pizza spots), "Bagel Boss" (5 delis), "World Eater" (10 different cuisine types), "Fine Diner" (5 fine dining), "Street Smart" (10 street food)

Culture Stamps

"Museum Rat" (5 museums), "Park Ranger" (10 parks), "Bridge Builder" (walk 3 bridges), "History Buff" (10 historic sites)

Social Stamps

"Tour Guide" (create a route others complete), "Tastemaker" (10 ratings with notes), "Night Owl" (5 spots after 10pm), "Early Bird" (5 spots before 9am)

Stamp Book Design

The stamp book is a dedicated page in the app — a visual grid of all possible achievements. Unearned stamps appear as grayed-out silhouettes (you can see what's possible but not the detail). Earned stamps are full-color with a satisfying reveal animation. Each stamp shows progress toward it ("3/5 pizza spots visited").

Implementation
1

Achievement Engine

A rules engine that listens for user actions (visit, rate, complete route) and checks against achievement criteria. Runs server-side to prevent cheating. Triggers push notification on unlock.

2

Stamp Artwork

Each stamp needs custom artwork — small illustrations in a consistent style. Could commission an artist or generate with AI art tools. The visual quality of stamps makes or breaks the feature.

3

Unlock Animations

When a stamp is earned: confetti burst, stamp "press" animation (like a rubber stamp), haptic feedback on mobile. The moment of earning a stamp should feel genuinely rewarding.

Social Features Phase 2

Friend networks + activity feed — explore together
Phase 2a: Friends & Activity Feed

Start with a friends list and activity feed. Users can add friends (by username, link, or contacts), see their activity (visits, stamps, routes completed), and compare progress. This creates accountability and friendly competition without the complexity of a full social network.

Friend Map Overlay

Toggle to see friends' visited spots on the map. Their spots appear as smaller dots in their profile color. "Ben visited Joe's Pizza last week" — social proof that drives exploration.

Activity Feed

Chronological feed: "Sarah earned 'Pizza Pilgrim'" / "Mike completed the Chinatown Crawl" / "Emma visited 3 new spots in Brooklyn". React with emoji. Tap to see details.

Leaderboards

Compare with friends: neighborhoods visited, stamps earned, spots rated. Weekly and all-time. Borough-specific leaderboards. Creates friendly competition.

Group Routes

Create a route and invite friends. Everyone's progress shows on the same map. "3/5 friends have completed this route." Shared experiences.

Phase 3: Expanded Social (Future)

Full profiles, following (beyond friends), photo reviews, route sharing to the community, user-curated "best of" lists, neighborhood ambassadors. This is the social network layer — build it only after the core experience is solid.

Transit & Live Navigation Phase 3

Full in-app navigation with real-time subway data
What "Full Navigation" Looks Like

This is the most ambitious feature. Full in-app navigation means: showing subway lines on the map, real-time train arrival data (MTA feeds), walking directions between stops, estimated travel times factoring in actual transit schedules, and turn-by-turn walking guidance. It's essentially building a mini Google Maps inside CTY, focused specifically on the NYC transit experience.

Data Sources for Transit
SourceDataCost
MTA GTFS-RT FeedReal-time subway arrivals, service alerts, delaysFree (API key required)
MTA Static GTFSSubway routes, stop locations, schedulesFree
Mapbox Directions APIWalking directions, route geometry, ETAsFree: 100K requests/month
Mapbox Navigation SDKTurn-by-turn guidance (mobile only)Free: 100K MAU (mobile SDK)
Implementation Phases
1

Phase 3a: Subway Overlay

Add MTA subway lines as a GeoJSON layer on the map. Color-coded by line. Show station dots. Tap a station to see lines that stop there. This alone makes the map massively more useful for navigation.

2

Phase 3b: Real-Time Arrivals

Tap a subway station to see real-time train arrivals (MTA GTFS-RT). "A train: 2 min, 8 min. C train: 5 min." Service alerts overlay. This is useful even without full navigation.

3

Phase 3c: Route Navigation

When following a Day Plan route, show the optimal path between stops. "Walk 5 min to 14th St station → L train 2 stops → Walk 3 min to destination." Combined walking + subway directions. Real-time ETA updates.

4

Phase 3d: Turn-by-Turn Walking (Mobile)

For mobile users, provide turn-by-turn walking navigation between spots using Mapbox Navigation SDK. "Turn left on Bedford Ave in 200ft." GPS tracking shows your position on the map.

Realistic Assessment

Full turn-by-turn navigation is a massive engineering effort typically handled by dedicated mapping companies. The pragmatic approach: build Phases 3a and 3b (subway overlay + real-time arrivals) which add enormous value with reasonable effort, then evaluate whether to build 3c/3d or integrate with Apple/Google Maps for the actual navigation piece.

Decision Log

Key choices made during planning
Map technology?
Mapbox GL JS — native rotation, custom styles, 3D, already used in Explorer page
Leaflet + CSS rotation has fundamental limitations. Mapbox GL handles bearing rotation natively, supports custom Studio styles, and the free tier covers 50K monthly loads.
Design direction?
Apple Maps-inspired quality + dynamic time-based + borough-adaptive theming
Clean, minimal, premium feel like Apple Maps. Theme shifts with time of day (5 periods) and subtly adapts to the borough being viewed. The city is alive — the app should reflect that.
Data sourcing strategy?
Multi-source aggregation: Google Places + Yelp + Foursquare + manual curation
No single API is complete. Cross-referencing multiple sources gives the most accurate, complete dataset. Start with top spots, expand to comprehensive coverage.
Data collection sequence?
Manhattan first (62 hoods) → Brooklyn → Queens → Bronx → Staten Island
Manhattan has the best boundary data and densest POI coverage. It's also where most early users will explore first. Process borough by borough systematically.
Route types?
Three types: curated, AI-generated, and user-created — all shareable
Curated routes are the flagship editorial content. AI generation handles personalization at scale. User-created routes become the community content engine.
Rating system?
Multi-dimension: Food/Experience, Vibe, Value, Service/Accessibility — each rated separately
A single star rating loses nuance. Multi-dimension captures "great food but bad service" scenarios. Different dimensions for restaurants vs attractions. Quick rate option for low friction.
Stamp system?
Achievement-based milestones across categories: Explorer, Foodie, Culture, Social
Simple check-ins aren't motivating enough. Achievement milestones ("visit 5 pizza spots") create goals that drive exploration. Categories ensure breadth of engagement.
Social scope?
Phase 2: Friends + activity feed. Phase 3: Full social platform.
Friends + feed is the 80/20 — most social value with manageable complexity. Full profiles, following, and community features come after the core experience is polished.
Transit integration depth?
Start with subway overlay + real-time arrivals. Evaluate full navigation later.
Subway overlay and real-time arrivals add enormous value with reasonable effort. Full turn-by-turn navigation is a massive engineering project — assess whether to build vs. hand off to Apple/Google Maps after Phase 3a/3b.
Timeline approach?
No rush, build it right. 3 phases over 10+ weeks. Foundation → Features → Scale.
Quality over speed. Each phase depends on the previous one being solid. The map and data must be excellent before layering on routes, social, and transit.
